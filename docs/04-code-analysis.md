# 04 — Разбор кода reclocking (путь A, без записи в железо)

Источники: реальные файлы upstream nouveau в `src/nvkm-clk/` и `src/nvkm-fb/`.
Маркировка: `[проверено по исходникам]` — прямо из кода (с file:line);
`[типовой паттерн]` — смысл регистра требует сверки с rnndb; `[?]` — гипотеза.

Версия: upstream master. Reclock-код NV50 меняется редко; при правке сверить с
исходниками **именно** ядра 7.0.11-cachyos (`srcversion 312143483E5151BCCE1C10D`).

## ГЛАВНЫЙ ВЫВОД: reclocking административно выключен на NV50 `[проверено по исходникам]`

Цепочка запрета при записи в debugfs `pstate` (и при boot-опции `NvClkMode`):

```
echo 0f > pstate
  → nvkm_clk_ustate()            base.c:521
  → nvkm_clk_ustate_update()     base.c:473
        if (!clk->allow_reclock)
                return -ENOSYS;   ← base.c:478-479  ВОЗВРАТ ОШИБКИ
```

А флаг для NV50 захардкожен в `false`:

```c
/* src/nvkm-clk/nv50.c:558-563 */
nv50_clk_new(...) {
    return nv50_clk_new_(&nv50_clk, device, type, inst, /*allow_reclock=*/false, pclk);
}
```

Для сравнения — родственный GT215 включает reclock:

```c
/* src/nvkm-clk/gt215.c:549 */
return nvkm_clk_ctor(&gt215_clk, device, type, inst, /*allow_reclock=*/true, &clk->base);
```

**Следствия:**
1. На G94 `echo 0f > pstate` возвращает **-ENOSYS** — переход не выполняется вообще
   (значит и «рискованный тест B» из прошлого шага просто упёрся бы в ошибку, не тронув железо).
2. `nouveau.config=NvClkMode=0f` тоже отвергается (та же проверка), а `auto`
   работает только при `allow_reclock` (`base.c:501`).
3. Карта остаётся на boot/POR-клоке (`AC: 500/1250/499`) — что мы и измерили.
4. Реализация reclock (`nv50_ram_calc/prog`, `nv50_clk_calc`) **полностью присутствует**
   и даже учитывает G94 особо (`ramnv50.c:441` «G94 does not even test these regs»),
   но намеренно заперта — upstream считает NV50 memory-reclock недостаточно проверенным.

## Оркестрация перехода pstate `[проверено по исходникам]`

`nvkm_pstate_prog()` (base.c:267):
```
1. nvkm_pcie_set_link()                       base.c:283  (ширина/скорость PCIe)
2. ЕСЛИ есть fb->ram->func->calc:              base.c:285
     do { ram->func->calc(khz); ram->func->prog(); } while(>0);  ← ПАМЯТЬ
     ram->func->tidy();
3. nvkm_cstate_prog(... HIGHEST)               base.c:296  → clk->calc + clk->prog  ← ЯДРО/ШЕЙДЕР
```
Выбор pstate (AC/DC, авто/ручной/темп/dstate-floor) — в `nvkm_pstate_work()` (base.c:300).

## Память: где именно может зависнуть (`nv50_ram_calc`, ramnv50.c) `[проверено по исходникам]`

Скрипт собирается в hwsq и исполняется в `nv50_ram_prog` (ramnv50.c:480) через
`ram_exec(..., NvMemExec)` — **опция `NvMemExec` (по умолч. true)**; при `NvMemExec=0`
скрипт строится, но в железо НЕ пишется (потенциально безопасная сухая прогонка).

Опасная последовательность (адреса — из исходника, семантику сверять с rnndb `memory/`):

| Шаг | Код | Риск |
|---|---|---|
| Блок FIFO | `0x002504=1` ramnv50.c:312 | — |
| **Disable FB** | `ram_setf(0x10,0x00)` + `ram_wait(0x00,0x01)` :314-315 | если FB не «уснёт» — зависание ожиданием |
| Precharge/refresh | `0x1002d4/0x1002d0` :321-323 | — |
| **Enter self-refresh** | `0x1002dc=1` :325 | VRAM теперь недоступна — любая ошибка дальше = чёрный экран |
| **Reprogram MPLL** | `nvbios_pll_parse(0x004008)`+`nv04_pll_calc` :327-334; запись `0x004008/0x00400c` :353-356 | неверные N/M/P → память не заведётся |
| Магические задержки | `ram_nsec 64000+32000` :363-364 (помечены `/*XXX*/`) | подобраны эмпирически, не из спеки |
| **Exit self-refresh** | `0x1002dc=0`,`0x1002d4=1`,`0x100210=0x80000000` :368-370 | momento истины: ретрейнинг |
| Перезапись таймингов | `timing[0..8]` :389-397 | неверный тайминг → corruption/висение |
| MR (mode regs) GDDR3 | :379-384, :453 | — |
| **Re-enable FB** | `ram_setf(0x10,0x01)`+`ram_wait(0x00,0x00)` :462-463 | если не поднимется — намертво |
| Разблок FIFO | `0x002504=0` :465 | — |

Тип памяти определяется по `0x100714 & 7` (ramnv50.c:552) → у нас GDDR3 (видно в
VBIOS и `nvkm_gddr3_calc`, ramnv50.c:292).

## Ядро/шейдер: `nv50_clk_calc` (nv50.c:368) `[проверено по исходникам]`

- Тоже блокирует FIFO и глушит FB на время (nv50.c:389-392, восстановление :489-491).
- Отсоединяет nvclk/sclk от их PLL перед перепрограммированием (chipset-зависимо :454-457).
- Core всегда через nvpll `0x4028` (:460-466); shader — либо привязка к nvclk,
  либо spll `0x4020` (:474-486).
- **Важная оговорка из кода:** shader должен быть **не меньше 2× core**, иначе
  «some chipsets will be very unhappy» (комментарий :468-473). Для 0f: core 650,
  shader 1625 → 1625 ≥ 1300 ✓ (запас есть).
- Чтение текущих частот (для проверки) — `nv50_clk_read` (:191), G94-ветки
  присутствуют (напр. :87-113, :261).

## Минимальный патч (кандидат) и его цена

**Размер — одна строка:** в `nv50_clk_new()` (nv50.c:562) передать `true` вместо `false`,
сняв гейт `-ENOSYS`. После этого уже существующие `calc/prog` начнут программировать
частоты по `echo 0f > pstate`.

**НО** это и есть ровно тот рискованный путь (memory reclock 499→900), который upstream
держит выключенным. Поэтому патч тривиален, а **валидация — главная работа**:

1. Безопасно: собрать с патчем + `nouveau.config=NvMemExec=0` → reclock-скрипт строится,
   но в память не пишет; проверить по `dmesg` (`nouveau.debug=clk=debug,fb=debug`),
   что `calc` проходит без `-EINVAL/-ERANGE` и тайминги печатаются (ramnv50.c:142-146).
2. Затем (рискованно, с планом восстановления): `NvMemExec=1`, поднять `0f`, замерить
   стабильность; при висении на memory-части — изолировать шаг по таблице выше.

**Чего НЕ делает:** не добавляет промежуточных pstate (их нет в VBIOS), не трогает вольтаж
(уже 1.15 В), не лечит TTM ENOMEM (это отдельный фронт, docs/03 §4).
