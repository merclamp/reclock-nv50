# 05 — Сборка пропатченного nouveau и БЕЗОПАСНАЯ прогонка

Состояние окружения (проверено):
- clang 22.1.6 + ld.lld 22.1.6 + llvm-objcopy — есть (ядро собрано clang+LTO).
- `CONFIG_MODULE_SIG_FORCE` нет, Secure Boot нет, lockdown=[none] ⇒ самосборный
  неподписанный модуль грузится (taint, безвредно).
- `CONFIG_MODVERSIONS` нет ⇒ важен только vermagic `7.0.11-1-cachyos`,
  получается автоматически при сборке против `/usr/lib/modules/$(uname -r)/build`.
- Пакет ядра: `linux-cachyos 7.0.11-1`.

## Шаг A. Получить исходники nouveau (единственный блокер)

Нужен полный каталог `drivers/gpu/drm/nouveau` версии, совпадающей с ядром 7.0.11.
Положить в `src/` (любой из путей: `src/linux`, `src/linux-source`, `src/nouveau`).
Поскольку MODVERSIONS выключен, незначительное расхождение версии не помешает
загрузке (vermagic), но для корректности лучше точная 7.0.11.

Варианты добычи (на выбор):
- distro-исходник cachyos (точное совпадение) — через PKGBUILD `linux-cachyos`;
- vanilla-тарбол соответствующей версии с kernel.org;
- git checkout нужного тега.

## Шаг B. Собрать (скрипт уже готов)

```
scripts/build-nouveau.sh [путь_к_исходникам]
```
Скрипт: копирует nouveau в `build/nouveau-src`, применяет
`patches/0001-...patch`, собирает `LLVM=1` против headers, печатает vermagic.
Ожидаемый результат: `build/nouveau-src/nouveau.ko` с vermagic `7.0.11-1-cachyos`.

## Шаг C. БЕЗОПАСНАЯ сухая прогонка (без записи в память!)

Цель: убедиться, что reclock-путь теперь открыт и `calc` считает частоты/тайминги
**без** реального переключения памяти.

1. Подготовить восстановление на всякий случай (см. §E) — даже для dry-run.
2. Выгрузить штатный модуль и загрузить собранный с отключённым исполнением памяти:
   ```
   sudo modprobe -r nouveau            # выгрузит при незанятой карте (выйти из X/Wayland!)
   sudo insmod build/nouveau-src/nouveau.ko \
        config="NvMemExec=0" debug="clk=debug,fb=debug,bios=debug"
   ```
   `NvMemExec=0` ⇒ hwsq-скрипт памяти строится, но в железо НЕ пишется (ramnv50.c:484).
3. Попробовать перейти в 0f и смотреть лог:
   ```
   echo 0f | sudo tee /sys/kernel/debug/dri/0000:01:00.0/pstate
   sudo dmesg | tail -40
   ```
   Успех dry-run: `nvkm_clk_ustate_update` больше НЕ возвращает -ENOSYS;
   в логе виден расчёт таймингов (`220:`/`230:`/`240:` из ramnv50.c:142-146)
   и подбор MPLL без `-EINVAL/-ERANGE`. Реальные частоты НЕ меняются (память не тронута).

## Шаг D. Боевой тест (РИСКОВАННО — отдельное согласие)

Только после успешного dry-run и с планом восстановления:
`NvMemExec=1` (по умолчанию), снова `echo 0f > pstate`, замерить:
`cat pstate` (применилось ли 900 МГц память), стабильность, артефакты.
При зависании на memory-части — изолировать по таблице docs/04 §«Память».

## E. План восстановления (обязателен перед C/D)

- Мы в графической сессии на ЭТОЙ карте ⇒ выгрузка/тест уронит сессию.
- Рекомендуется: зайти со второй машины по SSH и тестировать из текста (без X),
  либо иметь Magic SysRq (`echo 1 > /proc/sys/kernel/sysrq`; REISUB) для жёсткого ребута,
  либо просто готовность к hard-reset (reset-кнопка).
- Откат: `sudo rmmod nouveau; sudo modprobe nouveau` вернёт штатный модуль;
  при висении GPU — перезагрузка. Постоянных изменений патч не делает (только в ОЗУ модуля).
