# 03 — Reclocking: корневой анализ (по реальному VBIOS и dmesg)

Источник: дамп `traces/vbios.rom` (62976 б, BIOS `62.94.11.00.03`), разобран `nvbios`
из envytools. Полный разбор — `traces/nvbios-full.txt`. Всё ниже — `[измерено/декодировано]`.

## 1. Performance-таблица (PM_Mode at 0xcb36, v35)

Заявлено «4 performance levels», но реально заполнен **один**:

| pstate | Core | Memory | Shader | Voltage | Timing |
|---|---|---|---|---|---|
| `0f` | 650 MHz | 900 MHz | 1625 MHz | 1.15 В (115×10мВ) | 36 |
| (3 прочих) | 0 | 0 | 0 | — | — (пустые, ID `0xff`) |

**Вывод:** промежуточных pstate (07/0a) в VBIOS нет. Поэтому `pstate` в debugfs
показывает только `0f` и `AC` — это **свойство BIOS, а не баг драйвера**.

## 2. Voltage-таблица (at 0xce0a, v20) — VID через GPIO (VID_0/VID_1)

| Vid | Напряжение | GPIO (VID_0,VID_1) |
|---|---|---|
| 0x0 | 1.000 В | 0,0 |
| 0x1 | **1.150 В** | 1,0 |
| 0x2 | 1.100 В | 0,1 |

Измерено сейчас: `in0_input = 1150 мВ` ⇒ карта **уже на 1.15 В = напряжение уровня 0f**.

## 3. Ключевой вывод для проекта

- Текущий режим `AC: core 500 / shader 1250 / memory 499` — это безопасный
  boot/POR-клок; карта **не повышается** до `0f` сама.
- **Напряжение НЕ блокер** — оно уже на максимуме (1.15 В). Нужен только подъём частот.
- Самое ценное и самое опасное — **memory reclock 499 → 900 МГц** (×1.8):
  self-refresh → смена MPLL → ретайминг → выход из self-refresh. Любая ошибка → зависание.
- Задача проекта сузилась до: **сделать переход в `0f` (особенно memory) стабильным**,
  а не «добавлять pstate» и не «трогать вольтаж».

## 4. Сопутствующие дефекты на живой системе (dmesg) — отдельно от reclocking

| Счётчик | Сообщение | Кто | Трактовка |
|---|---|---|---|
| 30× | `ttm_validate … validate: -12` | Xwayland | **ENOMEM**: размещение BO в VRAM не удаётся (512 МБ давление/фрагментация) |
| 12× | `gr: DATA_ERROR [INVALID_VALUE] class 8297 mthd 0204` | Xwayland | плохое значение в 3D-классе NV50 |
| 2× | `fifo: CACHE_ERROR mthd 0060 data beef0201` | Renderer | подозрительный маркер `beef0201` |
| 1× | `fifo: DMA_PUSHER INVALID_CMD` | plasmashell | единичный сбой push-буфера |

Эти ошидки — кандидаты во второй фронт (TTM/eviction), но не смешивать с reclocking.
Текущее разрешение всего 1360×768, а VRAM 512 МБ ⇒ 30× ENOMEM подозрительны и
заслуживают отдельного разбора (возможен реальный баг eviction/placement).

## 5. Следующий шаг и его риск

Чтобы проверить, **поднимается ли `0f` стабильно или зависает на memory-части**,
нужен контролируемый тест `echo 0f > pstate`. Это потенциально вешает GPU.
Предусловие: способ восстановления (SSH с другой машины / magic SysRq / watchdog).
Безопасная альтернатива — сперва прочитать код `nvkm/subdev/fb/ramnv50.c`
(уже в `src/nvkm-fb/`) и понять последовательность, прежде чем дёргать железо.
