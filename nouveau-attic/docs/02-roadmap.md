# 02 — Роадмап (вертикальные срезы, на основе измерений)

Срезы 1–2 и 5 из промпта (KMS, modeset, GL-ускорение) **уже закрыты upstream** на этой
машине — не повторяем. Сосредотачиваемся на reclocking (срез 6) — единственном пробеле.

## Шаг 0 — Добыть полные исходники nouveau нашего ядра `[блокер]`

Headers есть, но `.c` nouveau нет. Варианты:
- `asp`/`pkgctl` исходники `linux-cachyos` (PKGBUILD тянет kernel tarball);
- зеркало upstream (kernel.org / github torvalds) — версия близкая к 7.0;
- envytools `rnndb` (XML регистров) — обязательно для проверки адресов.
- DoD: дерево `nvkm/subdev/clk` и `nvkm/subdev/fb` доступно для чтения; rnndb склонирован.

## Шаг 1 — Полная карта reclocking-состояния (read-only, безопасно)

- Прочитать все debugfs-узлы карты (`pstate`, и проверить имена `clk`/`volt`/`therm`).
- Снять `dmesg` nouveau при загрузке: есть ли отказы reclocking/volt/PMU.
- Понять, почему доступен только `0f` (нет 07/0a) — таблица в VBIOS или в `clk` subdev.
- DoD: документирован полный набор pstate, текущая логика выбора, причины отказа подъёма.

## Шаг 2 — Контролируемый тест подъёма pstate `[рискованно — нужно согласие]`

- **Предусловие:** способ удалённой перезагрузки (SSH с другой машины / sysrq / watchdog).
- `echo 0f > .../pstate`, замерить: частоты применились? стабильно? артефакты? зависание?
- Если зависает на memory-части — изолировать: пробовать только core/shader без memory.
- DoD: воспроизводимый вердикт «поднимается/зависает» + где именно (core vs memory).

## Шаг 3 — Локализовать дефект в исходнике

- Сопоставить точку отказа (шаг 2) с кодом `nvkm/subdev/fb/ram*` или `clk/*`.
- Сверить программируемые регистры с rnndb (`memory/`, `pll.xml`) — без догадок.
- DoD: конкретная функция/последовательность, где ломается, с цитатами строк.

## Шаг 4 — Минимальный патч

- Правка существующего кода (не новый модуль): корректная последовательность
  self-refresh / MPLL / retrain или фикс таблицы pstate / volt-маппинга.
- Сборка out-of-tree против `…/build` (headers есть), `modprobe` обновлённого nouveau.
- DoD по промпту: сигнатуры под 7.0, обработка ошибок и откат, блокировки,
  источник каждого регистра, команда проверки, явные ограничения.

## Шаг 5 — Игры (после стабильного reclocking)

- Поставить Gallium Nine + WineD3D, замерить FPS до/после стабильного `0f`.
- DoD: число (FPS) на конкретной D3D9-игре в low vs high pstate.

## Принципы (из промпта)

- Не выдумывать регистры — только rnndb/исходник.
- Маркировать уверенность каждого утверждения.
- Патч к существующему коду, не переписывание.
- Перед любой записью в железо — путь отката и согласие.
